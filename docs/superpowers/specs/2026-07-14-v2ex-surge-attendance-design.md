# V2EX Surge 每日签到设计

## 目标

在本仓库中提供一套独立的 V2EX 每日签到脚本与 Surge 模块。脚本不依赖原始参考仓库，只支持单个 Surge 账户，并且只有在 V2EX 每日任务页明确确认当天奖励已领取后才报告签到成功。

通知展示签到状态、连续天数、当天奖励和当前余额。连续天数、奖励或余额解析失败时可以省略对应字段，但不能影响签到成功状态的判断。

## 文件范围

- `scripts/v2ex/v2ex.js`：Cookie 捕获、每日任务读取、兑换、结果确认、重试、通知和脱敏日志。
- `modules/v2ex.sgmodule`：Surge HTTP request 与 cron 配置，支持自定义脚本地址、定时时间和调试开关。
- `tests/test_v2ex.mjs`：使用 Node.js 内置 `vm` 模拟 Surge 环境，覆盖签到状态机与敏感信息保护。

不增加第三方运行时依赖，不修改现有 NodeSeek 或 Tekqart 脚本。

## Surge 使用方式

模块提供以下参数：

- `SCRIPT_URL`：默认指向本仓库 `main` 分支中的 `scripts/v2ex/v2ex.js`。
- `CRON`：默认每天 09:00 执行。
- `DEBUG`：默认 `false`；启用后输出 HTTP 状态和经过脱敏、截断的响应摘要。

用户登录 `www.v2ex.com` 后访问个人页或每日任务页，HTTP request 脚本保存完整 Cookie 和当前 User-Agent。若 V2EX 登录 Cookie 没有变化，只更新持久化值和 User-Agent，不重复发送抓取成功通知。

## 签到流程

1. 从 Surge 持久化存储读取 Cookie 和 User-Agent。缺少 Cookie 时停止并提示用户重新抓取。
2. 请求 `https://www.v2ex.com/mission/daily`，使用已保存的 Cookie、User-Agent，并禁用缓存。
3. 解析任务页：
   - 页面明确要求登录时，报告 Cookie 失效。
   - 页面明确包含“每日登录奖励已领取”时，状态为“今日已签到”，不调用兑换接口。
   - 页面包含合法的 `/mission/daily/redeem?once=<数字>` 链接时，进入兑换流程。
   - 其他页面结构均报告“签到未确认”，不猜测成功。
4. 使用完整兑换链接发起 GET 请求，`Referer` 固定为每日任务页。
5. 无论兑换响应正文或状态看起来是否成功，都重新请求每日任务页。
6. 只有确认页明确包含“每日登录奖励已领取”才报告“签到成功”。确认页仍包含兑换链接或无法识别时，不报告成功。
7. 确认成功后请求 `/balance`，从最新的当天“每日登录奖励”流水提取奖励，并从余额区域提取金币、银币、铜币。解析不到时省略相应字段。

连续天数取自最终确认页，不能使用兑换前的数值冒充签到后的结果。

## 重试与错误分类

最多进行三次签到尝试，每次尝试都重新请求每日任务页并获取新的 `once`，不能复用旧值。

以下情况允许重试：

- 网络错误或超时。
- 兑换后任务页仍未显示已领取。
- 临时收到空页面、限流页或无法识别的 HTML。

以下情况立即停止：

- 页面明确要求登录。
- 第一次读取任务页即确认今日已经领取。
- 第三次尝试后仍无法确认。

最终状态分为：

- `签到成功`：本轮兑换后，任务页确认已领取。
- `今日已签到`：本轮开始前任务页已经确认已领取。
- `Cookie 失效`：任务页明确要求登录，或返回 401。
- `访问受限`：返回 403、429、503，或页面包含常见挑战/限流特征。
- `签到未确认`：没有任何可靠证据证明领取成功。
- `请求失败`：持续出现网络级错误。

HTTP 200、302、余额页可以访问、存在连续天数或兑换请求没有抛异常，都不能单独作为成功证据。

## 通知与日志

成功通知按可用信息组成：

```text
V2EX
签到成功
连续 1262 天 | 奖励 5 铜币 | 余额 3 金币, 56 铜币
```

重复签到使用“今日已签到”，失败状态使用明确原因。普通日志只记录流程、尝试次数和状态，不输出 Cookie 或完整 `once`。

调试日志可以记录请求标签、HTTP 状态、响应长度与最多 800 个字符的正文摘要，但必须脱敏：

- Cookie 认证值。
- URL 中的 `once`。
- HTML 中可能出现的 `once`。

## 测试标准

自动测试至少覆盖：

1. 捕获模式保存完整 Cookie 和 User-Agent。
2. Cookie 未变化时不重复通知，但仍刷新持久化数据。
3. 已签到页面不调用兑换接口并报告“今日已签到”。
4. 未签到页面提取动态 `once`，兑换后回查成功才报告“签到成功”。
5. 回归场景：兑换返回普通页面且余额页可访问，但任务页仍未领取时，绝不报告成功。
6. 每次重试使用任务页返回的新 `once`。
7. 三次均未确认时报告“签到未确认”。
8. 登录页和 HTTP 401 报告 Cookie 失效。
9. HTTP 403、429、503 或挑战页面报告访问受限。
10. 成功后解析连续天数、当天奖励与余额；字段缺失时仍保持已确认的成功状态。
11. 网络错误重试，并在耗尽次数后报告请求失败。
12. 调试日志和通知不泄露 Cookie 或 `once`。

测试不访问真实 V2EX，不需要保存用户凭据，所有响应均由 Surge 模拟器提供。

## 非目标

- 不支持 Loon 或 Quantumult X。
- 不支持多账户。
- 不自动登录，不保存用户名或密码。
- 不以余额差值推断签到成功。
- 不尝试绕过 V2EX 的访问控制或挑战页面。
